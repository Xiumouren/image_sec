## 2. 总体结论

当前 `source_engine` 已具备 MVP 级雏形，模块边界基本清楚，且已接入 `/api/detect` 主链路，能够返回 `source_analysis`。

但从架构质量、算法可信度和接口契约一致性来看，仍存在明显不足：

- `ELA` 模块当前实现可信度不足，是最主要问题。
- `/api/detect` 的降级策略没有完全符合设计文档。
- SQLite 与文件系统之间缺少一致性控制。
- `EXIF` 的语义判断不严谨。
- API 暴露了不必要的服务器绝对路径。
- 当前实现可运行，但仍偏“原型”，距离稳定的生产化后端模块还有差距。

## 3. 分模块审查

## 3.1 pHash 模块

**文件**
- [backend/source_engine/phash/phash_service.py](/C:/Code/net_sec/backend/source_engine/phash/phash_service.py)

**结论**
- 基本可用
- 存在性能和扩展性不足
- 与设计方向基本一致

**符合点**
- 能计算图片 `pHash`
- 能计算历史图片的 Hamming 距离
- 能返回 `matched`、`top_match`、`candidates`
- 响应结构总体接近设计文档中的 `phash` 结构

**存在不足**
1. 采用全量历史资产加载后逐条比对，扩展性差
2. 没有结合 `sha256` 做重复文件快速命中
3. 没有对坏数据或非法 `phash` 值做隔离
4. 阈值为固定经验值，缺少校准依据

**原因与情况**
- [repository.py:16](/C:/Code/net_sec/backend/storage_layer/sqlite/repository.py:16) 会读取全部资产记录
- [phash_service.py:26](/C:/Code/net_sec/backend/source_engine/phash/phash_service.py:26) 在 Python 层逐条计算距离
- 历史图数量增加后，接口耗时将线性增长

**风险**
- 资产规模增长后，`/api/detect` 性能明显下降
- 重复文件会持续重复归档、重复入库
- 单条异常 `phash` 数据可能影响整次分析

**修正建议**
- 增加 `sha256` 精确命中逻辑，作为第一层去重
- `pHash` 只负责相似度检索，不承担精确唯一识别
- 增加对历史脏数据的容错
- 后续可增加候选筛选机制，避免全表扫描

## 3.2 EXIF 模块

**文件**
- [backend/source_engine/exif/exif_service.py](/C:/Code/net_sec/backend/source_engine/exif/exif_service.py)

**结论**
- 具备基础提取能力
- 语义定义不严谨
- 存在误判与隐私暴露风险

**符合点**
- 能读取 EXIF
- 能输出摘要字段 `datetime_original / make / model / software`
- 能输出原始字段 `raw`

**存在不足**
1. `has_exif = bool(raw)` 判定过于粗糙
2. `edited_hint = bool(summary.get("software"))` 逻辑过于武断
3. `raw` 原始字段直接对外输出，存在隐私风险
4. 没有区分：
- 无 EXIF
- EXIF 被清洗
- EXIF 解析失败

**原因与情况**
- [exif_service.py:22](/C:/Code/net_sec/backend/source_engine/exif/exif_service.py:22) 到 [exif_service.py:30](/C:/Code/net_sec/backend/source_engine/exif/exif_service.py:30) 只要有任何 tag 就认为存在 EXIF
- [exif_service.py:28](/C:/Code/net_sec/backend/source_engine/exif/exif_service.py:28) 把 `Software` 字段存在直接视为编辑提示，这个判断证据强度不足

**风险**
- 正常手机照片、平台转码图被误判为“有编辑痕迹”
- 若后续图片含 GPS 等字段，可能对前端暴露敏感信息

**修正建议**
- `has_exif` 改为基于核心字段集判断
- `edited_hint` 改为更中性的字段，如：
- `software_present`
- `possible_postprocess_hint`
- `raw` 默认不全量返回前端，只保留白名单字段
- 若存在地理位置信息，默认脱敏

## 3.3 ELA 模块

**文件**
- [backend/source_engine/ela/ela_service.py](/C:/Code/net_sec/backend/source_engine/ela/ela_service.py)

**结论**
- 当前实现存在明显不足
- 算法统计口径不正确
- 是本次审查中最需要优先修正的模块

**符合点**
- 能生成 ELA 图
- 能输出：
- `is_tampered`
- `anomaly_ratio`
- `max_error_level`
- `description`
- `output_relative_path`

**核心不足**
1. `anomaly_ratio` 基于增强后的 ELA 可视化图计算，而不是原始差分图
2. 对所有输入格式统一转 JPEG 再分析，算法前提不成立
3. 当前阈值没有结合格式、质量、来源链路做校准

**原因与情况**
- [ela_service.py:39](/C:/Code/net_sec/backend/source_engine/ela/ela_service.py:39) 到 [ela_service.py:45](/C:/Code/net_sec/backend/source_engine/ela/ela_service.py:45)
- 代码先根据 `max_error_level` 对图像做亮度增强，再用增强后的图计算 `anomaly_ratio`
- 这会把微弱差异整体放大，导致比例统计失真
- 对 PNG/BMP/WebP 输入，JPEG-ELA 的业务含义本身就不稳定

**风险**
- 正常压缩图可能被误判为“篡改”
- 平台转码、截图、格式转换带来的压缩差异会被误当成局部编辑痕迹
- 前端若把 `is_tampered = true` 当成强结论，会产生误导

**修正建议**
- 统计分析和可视化增强分离
- `anomaly_ratio` 必须基于原始差分图而不是增强图
- ELA 正式判定只建议对 `jpg/jpeg` 启用
- 对 `png/bmp/webp` 返回降级结果，例如：
- `supported_for_detection = false`
- `unsupported_reason = "ELA tamper scoring is only reliable for JPEG-like inputs"`
- 将 `is_tampered` 改为更保守语义，如：
- `suspicious_compression_pattern`
- 用样本集重新校准阈值

## 4.1 SQLite 层

**文件**
- [backend/storage_layer/sqlite/db.py](/C:/Code/net_sec/backend/storage_layer/sqlite/db.py)
- [backend/storage_layer/sqlite/repository.py](/C:/Code/net_sec/backend/storage_layer/sqlite/repository.py)

**结论**
- 能完成基本持久化
- 但事务性、一致性和并发能力不足

**符合点**
- 有 `source_assets` 表
- 有 `source_analysis_records` 表
- 有索引：
- `sha256`
- `phash`
- `asset_id`

**存在不足**
1. `save_asset` 和 `save_analysis_record` 为两次独立写入
2. 缺少单次检测事务封装
3. 没有显式并发配置，如：
- `timeout`
- `WAL`
- busy retry
4. 没有唯一约束，重复文件会持续累积
5. `EXIF/ELA` 摘要仅作为 JSON 保存，不利于检索和统计

**原因与情况**
- [source_orchestrator.py:41](/C:/Code/net_sec/backend/source_engine/source_orchestrator.py:41) 与 [source_orchestrator.py:63](/C:/Code/net_sec/backend/source_engine/source_orchestrator.py:63) 是两次独立数据库写操作
- [db.py:44](/C:/Code/net_sec/backend/storage_layer/sqlite/db.py:44) 使用默认连接模式，没有额外并发增强配置

**风险**
- 出现：
- 资产已入库但分析记录未写入
- 数据库写一半失败
- 多请求时 `database is locked`
- 后续做统计和历史查询时会遇到脏数据和性能问题

**修正建议**
- 新增单事务方法，统一完成：
- 保存资产
- 保存分析记录
- 开启 SQLite `timeout`
- 启用 `WAL`
- 为 `sha256` 增加唯一约束或逻辑唯一控制
- 将常用摘要字段结构化落库，而不是全部压入 JSON

## 4.2 文件系统层

**文件**
- [backend/storage_layer/filesystem/asset_manager.py](/C:/Code/net_sec/backend/storage_layer/filesystem/asset_manager.py)

**结论**
- 目录与命名策略基本清楚
- 但缺少重复文件治理和失败补偿机制

**符合点**
- 原图与 ELA 图分目录管理
- 按日期分层
- 文件命名带时间戳与 UUID，避免覆盖
- 能生成相对路径和访问 URL

**存在不足**
1. 重复文件仍会重复归档
2. 文件复制成功但数据库失败时，会形成孤儿文件
3. 数据库成功、ELA 失败或后续失败时，会形成半完整记录
4. 缺少清理机制和巡检机制

**原因与情况**
- [asset_manager.py:47](/C:/Code/net_sec/backend/storage_layer/filesystem/asset_manager.py:47) 每次都会复制图片归档
- 当前没有围绕“归档”和“数据库提交”做一致性协调

**风险**
- 长期运行后，文件系统与数据库状态不一致
- 存储空间浪费
- 历史资产难维护

**修正建议**
- 先计算 `sha256`，再决定是否需要归档
- 为归档与落库添加补偿逻辑
- 增加孤儿文件巡检脚本
- 可以引入“临时归档 -> 事务成功后正式提交”的两阶段思路

## 5. `/api/detect` 响应结构审查

**文件**
- [backend/input_layer/api/app.py](/C:/Code/net_sec/backend/input_layer/api/app.py)
- [backend/input_layer/api/schemas.py](/C:/Code/net_sec/backend/input_layer/api/schemas.py)

- 单次调用返回完整 `source_analysis`
- `phash/exif/ela` 三块都在主响应中
- 任一子模块失败时，返回对应 `error` 字段
- 不应因为某一子模块失败导致整个检测失败

**当前结论**
- 部分符合
- 但没有完全对齐设计

**符合点**
- `source_analysis` 已接入 `DetectionResponse`

- `phash`
- `exif`
- `ela`
- 子模块结构中保留了 `error` 字段

**不符合点**
1. 编排器整体失败时，`source_analysis = None`
2. 暴露了服务器绝对路径：
- `original_image_path`
- `ela.output_path`
3. 接口暴露的字段比设计需要更多，且部分字段对前端无必要
4. 当前无法清晰表达“分析成功但存储失败”的状态

**原因与情况**
- [app.py:135](/C:/Code/net_sec/backend/input_layer/api/app.py:135) 到 [app.py:142](/C:/Code/net_sec/backend/input_layer/api/app.py:142) 中，一旦 `source_orchestrator.analyze()` 抛异常，就把整个 `source_analysis` 设为 `None`
- [schemas.py:56](/C:/Code/net_sec/backend/input_layer/api/schemas.py:56) 和 [schemas.py:39](/C:/Code/net_sec/backend/input_layer/api/schemas.py:39) 定义了绝对路径字段

**风险**
- 与设计文档不一致
- 前端难以区分：
- source 分析失败
- source 存储失败
- 某个子模块失败
- 绝对路径暴露了服务端内部部署细节

**修正建议**
- 即使归档或持久化失败，也尽量返回完整 `source_analysis`
- 删除绝对路径，仅保留：
- `relative_path`
- `output_relative_path`
- `output_url`
- 建议新增 `storage` 子块，表达持久化状态

## 6. 潜在风险汇总

## 6.1 算法风险
- `ELA` 可能大幅误报
- `EXIF.edited_hint` 可能大幅误判

## 6.2 数据一致性风险
- 文件已归档但数据库未入库
- 资产入库成功但分析记录缺失
- 数据库与文件系统长期不一致

## 6.3 并发与稳定性风险
- SQLite 在多请求下可能发生锁冲突
- 缺少事务管理导致失败边界不清晰

## 6.4 安全与隐私风险
- 暴露服务器本地绝对路径
- 可能直接输出敏感 EXIF 元数据

## 6.5 可扩展性风险
- `pHash` 全表扫描会拖慢 `/api/detect`
- JSON 化存储不利于后续查询和运营分析

## 8. 建议的响应结构修正版

建议将 `source_analysis` 设计为“分析结果”和“存储状态”分离：

```json
{
  "source_analysis": {
    "phash": {
      "hash": "ff00aa...",
      "matched": false,
      "top_match": null,
      "candidates": [],
      "error": null
    },
    "exif": {
      "has_exif": true,
      "summary": {
        "datetime_original": "2024:01:01 10:00:00",
        "make": "Apple",
        "model": "iPhone 13",
        "software": "iOS"
      },
      "raw": {},
      "edited_hint": false,
      "error": null
    },
    "ela": {
      "is_tampered": false,
      "anomaly_ratio": 1.2,
      "max_error_level": 12,
      "description": "No strong localized compression anomalies were detected.",
      "output_relative_path": "ela/20260423/example_ela.jpg",
      "output_url": "/source-assets/ela/20260423/example_ela.jpg",
      "error": null
    },
    "storage": {
      "archived": true,
      "record_saved": true,
      "error": null
    }
  }
}
```

这样可以明确表达：

- 分析是否成功
- 文件是否归档成功
- 数据库是否写入成功
- 哪个子模块失败
- 前端应该如何降级展示

## 9. 最终结论

当前实现具备 MVP 原型形态，模块组织方向基本正确，但存在以下关键不足：

- `ELA` 结果不可靠
- `EXIF` 语义过强且可能误导
- SQLite 与文件系统缺乏事务一致性
- `/api/detect` 的 `source_analysis` 降级行为不符合设计要求
- API 暴露绝对路径与原始敏感信息的风险较高

因此当前版本可以视为：

> 可运行的溯源原型，但不建议直接作为可信的生产级溯源结论模块使用。

在进入下一阶段前，建议至少先完成以下四项修正：

1. 修正 ELA 统计逻辑
2. 修正 `/api/detect` 的降级与响应契约
3. 增加 SQLite 事务与失败补偿
4. 收敛 EXIF 与路径暴露语义
