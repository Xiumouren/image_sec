# 图片检测与溯源流程说明

本文说明当前项目中单图 `/api/detect` 是如何完成“图片检测 + 来源溯源”的。这里的“溯源”不是单一算法给结论，而是把 SHA-256、pHash、本地 CLIP + FAISS、EXIF、ELA、归档记录等信号组织成候选来源的证据链，再给出可信等级和人工复核建议。

## 1. 总体链路

单图检测的主入口是 `backend/input_layer/api/app.py` 中的 `detect()`。

一次上传图片后的主流程如下：

```text
前端上传图片
  -> FastAPI /api/detect
  -> 保存临时上传文件到 outputs/uploads/single/
  -> TensorFlow NSFW 分类
  -> Grad-CAM / saliency / integrated gradients 解释图
  -> SourceOrchestrator 溯源证据链分析
  -> 可选 Qwen3-VL-Plus 大模型安全复核
  -> 写 single report
  -> 返回 DetectionResponse
```

返回结果中的核心字段：

- `predicted_label` / `scores`：NSFW 分类结果
- `explanation`：解释图路径和方法
- `source_analysis`：溯源证据链
- `llm_safety_analysis`：外部多模态大模型复核结果

## 2. 图片检测阶段

后端先做基础校验和保存：

1. `validate_extension()` 检查文件扩展名，只允许 `.jpg`、`.jpeg`、`.png`、`.bmp`、`.webp`。
2. `save_upload()` 将文件保存到 `outputs/uploads/single/`。
3. `detector_service.predict(image_path)` 调用本地 TensorFlow 模型预测。
4. 根据用户选择的方法生成解释图：
   - `gradcam`
   - `saliency`
   - `integrated_gradients`

解释图生成失败不会中断检测，错误会写入 `explanation_error`。

随后后端构造 `detection_payload`，把图片名、预测类别、分数和解释图信息交给溯源模块。

## 3. 溯源模块入口

溯源入口是 `backend/source_engine/source_orchestrator.py` 中的：

```python
SourceOrchestrator.analyze(image_path, detection_payload)
```

这个方法负责串联：

- SHA-256 精确归档匹配
- pHash 感知哈希匹配
- EXIF 元数据提取
- ELA 压缩异常分析
- 本地 CLIP embedding 生成
- FAISS 语义候选召回
- 归档图片和数据库写入
- 候选证据合并、可信度判断和审核建议生成

## 4. SHA-256 与归档资产复用

第一步计算当前上传图片的 SHA-256：

```python
source_sha256 = asset_manager.compute_sha256(image_path)
existing_asset = repository.find_asset_by_sha256(source_sha256)
```

如果数据库中已有相同 SHA-256：

- 认为这是同一份文件的再次上传
- 复用历史 `source_assets` 记录
- 复用历史归档路径和 pHash
- `asset_reused = true`

如果没有命中：

- 稍后将图片复制到归档目录
- 写入新的 `source_assets` 记录
- `asset_reused = false`

SHA-256 是当前证据链中最强的来源信号，因为它代表文件内容完全一致。

## 5. pHash 感知哈希分析

pHash 由 `backend/source_engine/phash/phash_service.py` 负责。

如果当前图片已存在归档记录，直接复用历史 pHash；否则重新计算：

```python
current_phash = existing_asset.phash if existing_asset else phash_service.compute_hash(image_path)
```

随后从 SQLite 查询除当前 SHA-256 之外的历史资产：

```python
historical_assets = repository.list_assets_except_sha256(source_sha256)
```

pHash 会逐个比较历史图片：

- 将历史 pHash 转成 imagehash 对象
- 计算 Hamming distance
- 计算派生相似度：`1 - distance / total_bits`
- 按距离升序、相似度降序排序
- 只保留 Top K

当前默认阈值：

- `PHASH_MATCH_THRESHOLD = 8`
- `PHASH_TOP_K = 5`

注意：pHash 是“感知哈希信号”，用于发现构图或像素感知上接近的图片。它不是语义理解，也不能单独证明来源一致。因此新前端不再把 pHash 百分比展示成“视觉相似度结论”，而是展示为 `pHash distance` 和证据项。

## 6. EXIF 元数据分析

EXIF 由 `backend/source_engine/exif/exif_service.py` 负责。

它会读取图片中的白名单字段：

- `EXIF DateTimeOriginal`
- `Image Make`
- `Image Model`
- `Image Software`
- `Image Artist`
- `Image Orientation`

输出结构包括：

- `has_exif`
- `summary`
- `raw`
- `software_present`
- `possible_postprocess_hint`
- `error`

其中 `possible_postprocess_hint` 会检查软件字段是否包含 Photoshop、Lightroom、GIMP、Canva、Snapseed 等后处理软件线索。

EXIF 只作为辅助上下文。没有 EXIF 不代表图片可疑，有 EXIF 也不代表来源可信。

## 7. ELA 压缩异常分析

ELA 由 `backend/source_engine/ela/ela_service.py` 负责。

当前只对 JPEG 类输入做有效评分：

- `.jpg`
- `.jpeg`

非 JPEG 图片会返回降级结果：

```text
ELA scoring was skipped because the input is not JPEG-like.
```

JPEG 图片的处理过程：

1. 将原图转为 RGB。
2. 用固定 JPEG quality 重新压缩。
3. 对比原图和重压缩图的像素差。
4. 保存 ELA 可视化图到归档资产目录。
5. 计算高差异像素比例 `anomaly_ratio`。
6. 根据阈值判断 `is_tampered`。

当前默认参数：

- `ELA_JPEG_QUALITY = 90`
- `ELA_PIXEL_THRESHOLD = 24`
- `ELA_TAMPER_RATIO_THRESHOLD = 5.0`

ELA 只提示压缩异常，不直接证明图片被篡改。它更适合作为人工复核提示。

## 8. 本地 CLIP embedding 生成

语义向量由 `backend/source_engine/clip_feature/clip_embedding_service.py` 负责。

默认配置：

```text
LOCAL_CLIP_MODEL_NAME = ViT-B/32
device = cpu
```

处理过程：

1. 延迟加载本地 `clip` 包和 `torch`。
2. 调用 `clip.load("ViT-B/32", device="cpu")`。
3. 使用 CLIP preprocess 处理图片。
4. 调用 `model.encode_image()` 得到图片向量。
5. 转成 `float32`。
6. 做 L2 normalize。

归一化非常关键，因为后续 FAISS 使用 inner product。对于 L2 normalized 向量，inner product 等价于 cosine similarity。

如果 `clip`、`torch`、模型权重或图片处理失败，错误会被 `SourceOrchestrator._safe_semantic_search()` 捕获，写入：

```text
source_analysis.errors
signals.semantic_retrieval.error
```

主检测流程不会因此失败。

## 9. FAISS 语义候选召回

语义检索由 `backend/source_engine/semantic_retrieval.py` 中的 `SemanticRetrievalService` 负责。

当前索引策略：

- 使用 `faiss.IndexFlatIP`
- 不持久化 `.index` 文件
- 服务启动后首次请求时，从 SQLite 的 `source_assets` 重建内存索引
- 用独立数组 `_asset_by_row` 保存 `faiss_row -> source_asset.id`
- 用 `_assets_by_id` 保存 `asset_id -> SourceAssetRecord`

这样不会依赖 SQLite 行顺序，也避免 `.index` 文件和数据库映射不同步。

### 9.1 首次检索时重建索引

第一次调用 `search()` 时会触发：

```python
self._ensure_index()
self.rebuild_index()
```

`rebuild_index()` 会：

1. 从 SQLite 读取所有 `source_assets`。
2. 对每个资产读取或补算 embedding。
3. 如果 embedding 文件已存在，直接 `np.load()`。
4. 如果 embedding 缺失，则用 CLIP 对归档图片补算。
5. 将 embedding 加入 FAISS。
6. 同步更新 `_asset_by_row` 和 `_assets_by_id`。

历史资产缺少 embedding 时，会在重建索引时自动补算。

### 9.2 当前上传图片检索

当前上传图片会先生成 query embedding，然后调用：

```python
distances, indices = index.search(query_embedding.reshape(1, -1), top_k)
```

返回的候选会包含：

- `asset_id`
- `image_name`
- `relative_path`
- `source_url`
- `semantic_similarity`
- `model_name`

默认 Top K：

```text
SEMANTIC_RETRIEVAL_TOP_K = 5
```

## 10. embedding 文件与数据库映射

SQLite 中新增了 `source_asset_embeddings` 表，用来记录 embedding 元数据：

- `asset_id`
- `model_name`
- `embedding_dim`
- `embedding_path`
- `created_at`

真实向量不直接塞进 SQLite，而是保存为 `.npy` 文件：

```text
SEMANTIC_EMBEDDING_STORE_DIR/
  clip_ViT-B_32/
    asset_1.npy
    asset_2.npy
```

写入逻辑：

1. `_load_or_create_embedding()` 先查 `source_asset_embeddings`。
2. 如果记录存在且文件存在，读取 `.npy`。
3. 如果不存在，调用 CLIP 生成 embedding。
4. `np.save()` 保存文件。
5. `repository.upsert_embedding()` 写入或更新 SQLite 映射。

## 11. 当前图片归档与增量索引

如果当前图片不是历史资产，`AssetManager.archive_image()` 会将其复制到：

```text
SOURCE_ASSETS_ROOT/images/YYYYMMDD/HHMMSS_xxxxxxxx_original_name.ext
```

随后 `repository.persist_analysis()` 会写入：

- `source_assets`
- `source_analysis_records`

写库成功后，调用：

```python
semantic_service.add_or_update_asset(persist_result.asset, archived_asset.archived_path)
```

这会：

1. 为新归档图片生成或读取 embedding。
2. 写入 `source_asset_embeddings`。
3. 如果内存 FAISS 索引已存在，将该 embedding 增量加入索引。
4. 更新 `faiss_row -> asset_id` 映射。

如果当前图片是重复 SHA-256，服务不会重复加入同一个 asset id。

## 12. 候选证据合并

`SourceOrchestrator._build_evidence_chain()` 会调用 `_merge_candidates()`，把两个来源的候选合并到同一个候选列表：

- 语义检索候选：来自 CLIP + FAISS
- pHash 候选：来自感知哈希距离排序

合并键是 `asset_id`。

每个候选会形成类似结构：

```json
{
  "asset_id": 1,
  "image_name": "example.jpg",
  "relative_path": "images/20260515/example.jpg",
  "source_url": "/source-assets/images/20260515/example.jpg",
  "semantic_similarity": 0.86,
  "phash_distance": 10,
  "phash_similarity": 0.84,
  "exact_sha256_match": false,
  "evidence": [],
  "credibility_level": "medium",
  "needs_human_review": false
}
```

然后 `_candidate_evidence()` 会生成候选级证据项。

## 13. 候选证据项

当前证据项主要有三类：

### 13.1 `semantic_clip`

来自 CLIP + FAISS 的语义相似度。

分级规则：

- `strong`：`semantic_similarity >= 0.88`
- `medium`：`semantic_similarity >= 0.60`
- `weak`：低于 `0.60`

### 13.2 `perceptual_hash`

来自 pHash 的距离和派生相似度。

分级规则：

- `strong`：`phash_distance <= 6`
- `medium`：`phash_distance <= 12`
- `weak`：大于 `12`

消息中会明确说明这是 perceptual hash signal，不是 semantic proof。

### 13.3 `sha256`

如果 SHA-256 完全一致，则加入强证据：

```text
SHA-256 exactly matches an archived asset.
```

这是当前最强的来源一致性证据。

## 14. 候选可信度规则

候选级可信度由 `_candidate_credibility()` 计算。

当前规则：

1. 如果 `exact_sha256_match = true`：
   - `high`
2. 如果 CLIP 相似度高且 pHash 距离很小：
   - `semantic >= 0.88` 且 `phash_distance <= 8`
   - `high`
3. 如果 CLIP 很高但 pHash 很差：
   - `semantic >= 0.88` 且 `phash_distance > 18`
   - `needs_human_review`
4. 如果 pHash 很强但语义很低：
   - `semantic < 0.72` 且 `phash_distance <= 6`
   - `needs_human_review`
5. 如果 CLIP 较高：
   - `semantic >= 0.60`
   - `medium`
6. 如果只有中等证据项：
   - `medium`
7. 如果只有弱证据：
   - `low`
8. 没有证据：
   - `unknown`

这套规则的重点是：不让单一弱信号直接给出强归因；当语义和 pHash 冲突时，优先推给人工复核。

## 15. 全局可信度与人工复核

候选列表生成后，会按以下顺序排序：

1. `credibility_level`
2. `semantic_similarity`
3. `phash_similarity`

全局 `source_credibility_level` 由 `_overall_credibility()` 计算。

规则概览：

- 没有候选：`unknown`
- 有错误且没有候选：`unknown`
- 任一候选需要人工复核：`needs_human_review`
- Top 1 和 Top 2 的语义分数非常接近，差值小于 `0.03`：`needs_human_review`
- 否则使用 Top candidate 的可信等级

对应的 `review_recommendation` 会根据等级输出审核建议，例如：

- `high`：可将 Top candidate 作为强来源匹配，但仍需核验上下文和策略类别
- `medium`：人工查看 Top candidate 和元数据后再归因
- `low`：只能作为弱线索
- `needs_human_review`：证据强但冲突或候选接近，必须人工比对
- `unknown`：当前图库或检索无法支持可靠归因

## 17. 前端如何展示溯源

前端组件是：

```text
frontend/presentation_layer/src/components/SourceAnalysisCard.vue
```

展示内容：

- 当前图片：
  - SHA-256
  - pHash
  - 是否已归档
  - embedding 模型、维度、索引资产数量
- Top candidates：
  - 候选缩略图
  - 归档图链接
  - 语义相似度
  - pHash 距离
  - SHA-256 是否完全一致
  - 证据标签
  - 候选可信等级
- 人工复核提示：
  - 当 `source_credibility_level = needs_human_review` 时显示 warning
- 证据摘要与降级错误

前端刻意不再把 pHash 百分比包装成“视觉相似度”，而是明确显示 pHash 距离和证据类型。

## 18. 大模型复核如何使用溯源结果

大模型复核由：

```text
backend/source_engine/llm_review/llm_safety_review_service.py
```

负责。

它会把 `detection_payload` 和完整 `source_analysis` 一起放进请求 JSON，再把图片以 data URL 形式传给多模态模型。

大模型输出：

- `content_risk_level`
- `pornographic_assessment`
- `source_credibility_level`
- `source_credibility_assessment`
- `evidence_summary`
- `review_recommendation`
- `limitations`

注意：大模型只是解释辅助，不是唯一归因依据。真正的候选召回和证据结构来自本地溯源模块。

## 19. 降级与失败处理

当前设计要求：溯源模块失败不阻断 NSFW 检测。

几个典型降级点：

- 解释图生成失败：返回分类结果，`error` 中带解释图错误
- pHash 失败：`signals.phash.error`
- EXIF 失败：`signals.exif.error`
- ELA 失败：`signals.ela.error`
- CLIP / torch / faiss 失败：`signals.semantic_retrieval.error`
- embedding 归档失败：`query.embedding.archive_status.error`
- SQLite 写入失败：回滚，并删除本次新归档文件和 ELA 图
- 大模型 API Key 缺失：`llm_safety_analysis.error`

如果溯源总编排抛出异常，`/api/detect` 会返回一个最小可用的 `source_analysis`：

```text
source_credibility_level = unknown
candidates = []
errors = ["Source analysis orchestration failed: ..."]
```

## 20. 当前局限性

当前实现是第一版证据链溯源，仍有以下限制：

- CLIP 相似度阈值是经验值，还没有基于真实数据集校准。
- FAISS 使用内存索引，服务重启后会重建，适合当前规模，不适合超大图库。
- 当前未持久化 `.index` 文件，避免索引文件和 SQLite 映射不同步。
- EXIF 和 ELA 只是辅助信号，不能作为硬证据。
- 批量检测接口尚未接入完整证据链。
- 相似图片、裁剪图片、二次压缩图片的归因仍需要人工复核。
- 目前没有将候选间冲突、EXIF 冲突、ELA 异常全部细化进候选级证据，后续可以扩展。

## 21. 一句话总结

当前溯源系统的核心思想是：

```text
先用本地 CLIP + FAISS 找可能来源，再用 SHA-256、pHash、EXIF、ELA 组织证据链，最后根据多信号一致性给出可信等级和人工复核建议。
```

它不是“自动判定来源”的单点模型，而是面向审核员的证据聚合与候选召回系统。
